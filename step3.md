# Spring Core 학습 정리: step3 (빈 스코프 & 생명주기 콜백)

이 문서는 컴포넌트 스캔을 활용한 자동 빈 등록부터 스프링 빈의 대표적인 두 가지 스코프(싱글톤과 프로토타입), 그리고 빈의 생명주기(Lifecycle) 콜백 메커니즘을 정리한 학습 가이드입니다.

---

## 1. 🐣 초심자를 위한 비유
식당(스프링 컨테이너)에서 손님들에게 편의 도구를 제공해주는 서비스 상황에 빗대어 이해해 봅시다.

### 🚰 싱글톤 빈 (공용 정수기)
* 식당 홀에 딱 한 대 설치되어 있는 **정수기(SingletonBean)**입니다.
* 손님 A(sb1)가 물을 마시러 가도, 손님 B(sb2)가 물을 마시러 가도 모두 같은 정수기 인스턴스를 공유해서 사용합니다 (`sb1 == sb2 : true`).
* 매니저(컨테이너)는 식당 문을 열 때 정수기를 설치 및 작동 점검(`@PostConstruct`)하고, 식당 영업이 끝나 영업을 종료할 때 정수기 전원을 끄고 청소(`@PreDestroy`)하는 것까지 전 과정을 관리합니다.
* 여러 사람이 공유하므로, 누군가 정수기 온도를 바꿔놓거나 고장 내면 다른 사람에게도 영향이 갑니다. 따라서 싱글톤 객체는 상태(데이터)를 함부로 변경해서는 안 됩니다. (**무상태 설계 필요**)

### 🥤 프로토타입 빈 (일회용 종이컵)
* 손님이 물을 마시고 싶을 때마다 매번 새로 나눠주는 **종이컵(PrototypeBean)**입니다.
* 손님 A(pb1)가 받은 종이컵과 손님 B(pb2)가 받은 종이컵은 서로 다른 종이컵입니다 (`pb1 == pb2 : false`).
* 매니저(컨테이너)는 손님이 요청할 때만 새 종이컵을 꺼내서 전달(`@PostConstruct`)해 줍니다.
* 하지만 손님이 종이컵을 다 쓴 후에 그것을 구겨서 버리든 가져가든 매니저는 관여하지 않습니다. 즉, 다 쓴 종이컵을 회수하고 처리하는 것(`@PreDestroy`)은 매니저의 책임이 아니며, 오롯이 종이컵을 받은 손님(클라이언트)이 알아서 처리해야 합니다. (**컨테이너가 소멸 메서드를 호출하지 않음**)

## 2. 🛠️ 주니어를 위한 핵심 원리 설명

### 2.1. 컴포넌트 스캔 (@ComponentScan & @Component)
* step2에서는 `AppConfig`에 개발자가 `@Bean`을 작성해 직접 빈을 등록했습니다. 클래스가 많아질수록 설정 파일이 비대해집니다.
* step3에서는 `@ComponentScan`을 활용하여 특정 패키지 경로를 지정하면, 그 하위에 위치한 `@Component` 어노테이션이 붙은 클래스들을 스프링이 구동 시점에 클래스 패스를 스캔하여 자동으로 스프링 빈으로 등록합니다.

### 2.2. 빈 스코프 (Bean Scope)
* **싱글톤 (Singleton)**: 스프링 컨테이너의 시작부터 종료까지 단 하나의 인스턴스만 생성되어 관리되는 기본 스코프입니다.
* **프로토타입 (Prototype)**: 스프링 컨테이너에 빈을 요청할 때마다 새로운 객체를 생성하고 의존관계를 주입하여 반환하는 스코프입니다.

### 2.3. 빈 생명주기(Lifecycle) 콜백
스프링은 빈이 생성되고 의존관계 주입이 완료된 시점과, 스프링 컨테이너가 소멸하기 직전에 개발자가 초기화 및 정리 작업을 할 수 있도록 콜백 어노테이션을 지원합니다. (`jakarta.annotation-api` 의존성 필요)
1. **`@PostConstruct`**: 빈의 의존성 주입이 완료된 직후 실행되는 초기화 메서드입니다.
2. **`@PreDestroy`**: 컨테이너가 소멸하기 직전에 자원 반납 등을 위해 호출되는 소멸 메서드입니다.

> [!WARNING]
> **프로토타입 빈의 생명주기 한계**
> 프로토타입 빈은 스프링 컨테이너가 생성, 의존성 주입, 초기화 콜백(`@PostConstruct`)까지만 관여하고 클라이언트에 인스턴스를 전달한 이후에는 전혀 추적/관리하지 않습니다. 따라서 **컨테이너가 종료되더라도 `@PreDestroy`와 같은 소멸 콜백 메서드가 호출되지 않습니다.** 자원 반납 등의 소멸 처리는 해당 빈을 전달받은 클라이언트가 직접 수행해야 합니다.

### 2.4. Spring Web MVC에서의 웹 스코프 (Request, Session)
* 스프링은 프로토타입 외에도 웹 환경(Spring Web MVC 등)에서만 동작하는 **웹 스코프(Web Scope)**를 지원합니다.
  * **request 스코프**: HTTP 요청이 들어와서 나갈 때까지 유지되는 스코프입니다. 각각의 HTTP 요청마다 개별 빈 인스턴스가 동적으로 생성되고 할당됩니다.
  * **session 스코프**: 웹 세션의 시작부터 만료(소멸) 시점까지 유지되는 스코프입니다. 동일 사용자의 세션 내에서만 하나의 빈 인스턴스가 공유됩니다.
* **프로토타입과의 비교**: 매 요청마다 다른 객체를 생성한다는 점에서 프로토타입과 유사하지만, 웹 스코프 빈은 HTTP 요청 종료 또는 세션 만료 시 스프링 컨테이너가 소멸 콜백(`@PreDestroy`)까지 정상적으로 호출해 준다는 결정적인 차이가 있습니다.

### 2.5. 실무에서의 빈 스코프 활용 방식
* **실무에서는 실질적으로 99.9% 싱글톤(Singleton) 빈만 작성합니다.**
* 싱글톤 빈 내부에 프로토타입 빈을 주입하여 혼용해 쓰는 복잡한 구조는 동시성 오류 유발 가능성과 관리의 어려움 때문에 실무에서 거의 쓰이지 않습니다.
* 상태 저장이 필요한 데이터가 있다면 빈 내부에 저장하지 않고, **메서드의 파라미터, 지역 변수, DTO**에 담아 무상태(Stateless) 형태로 전달하는 설계 방식을 사용합니다.

## 3. 💬 면접을 위한 예상 질문 & 답변 (Q&A)

### Q1. `@Component`와 `@Bean` 어노테이션의 용도 및 차이점을 설명해 주세요.
> **Answer**
> * **`@Component`**는 클래스 레벨에 선언하며, 개발자가 직접 작성한 커스텀 클래스를 컴포넌트 스캔의 대상으로 지정하여 스프링이 자동으로 빈으로 등록하도록 할 때 사용합니다.
> * **`@Bean`**은 메서드 레벨에 선언하며, 주로 직접 코드를 수정할 수 없는 외부 라이브러리(3rd-party) 객체를 빈으로 등록하거나 특별한 초기화 로직이 필요해 수동으로 빈을 관리해야 할 때 `@Configuration` 클래스 내부에서 사용합니다.

### Q2. 싱글톤 빈과 프로토타입 빈의 생명주기 관리 범위의 결정적인 차이는 무엇인가요?
> **Answer**
> * **싱글톤 빈**은 스프링 컨테이너 기동 단계에 생성 및 초기화(`@PostConstruct`)되며, 컨테이너가 소멸하기 직전 소멸 콜백(`@PreDestroy`)까지 수행하여 컨테이너가 생명주기의 전 과정을 완전히 관리합니다.
> * **프로토타입 빈**은 컨테이너에 요청할 때마다 동적으로 생성되고 초기화(`@PostConstruct`)까지는 실행되지만, 일단 반환되고 나면 스프링 컨테이너가 관리 대상에서 제외합니다. 따라서 컨테이너 종료 시 소멸 콜백(`@PreDestroy`)이 작동하지 않는다는 차이가 있습니다.

### Q3. 실무에서도 싱글톤 빈과 프로토타입 빈을 함께 혼용하여 많이 사용하나요?
> **Answer**
> * **아닙니다. 실무에서는 사실상 99.9% 싱글톤 빈만을 사용해 시스템을 설계합니다.**
> * 싱글톤 내부에서 프로토타입 빈을 주입받아 사용하는 등의 복잡한 혼용 방식은 설계 복잡성을 높이고 동시성 버그를 초래할 수 있어 거의 사용하지 않습니다.
> * 비즈니스 로직 클래스는 철저히 **무상태(Stateless)**로 설계하며, HTTP 요청 단위의 상태 데이터가 필요한 경우에는 스프링의 `request`/`session` 웹 스코프를 활용하거나, 대부분 컨트롤러 단계의 DTO나 지역 변수를 사용하여 무상태 싱글톤 서비스 흐름 안에서 처리합니다.
